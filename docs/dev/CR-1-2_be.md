Наша следующая задача - обновить API-сервис и модель базы данных.

Мы уже внесли изименения в пользовательский интерфейс. Теперь необходимо оьновить API-сервис. Концептуально, мы не меняем поведение API-сервиса. Т.е. последовательность должна ыбть следующей:
1. Пользователь отправляет POST-запрос на эндпоинт с данными объявления (текст, изображения, ссылка).
2. API-сервис принимает запрос
3. Валидация запроса:
  - описание должно быть заполнено, это обязательная часть
  - изображения могу отсутствовать, но если они есть, то их должно быть не более 5 и размер каждого изображения должен быть не более 2560 кб
  - ссылка может отсутствовать, но если она есть, то она должна быть валидной и вести на сайт otomoto.pl
4. Если валидация прошла успешно, то API-сервис сохраняет данные пользовательского запроса в базу данных.
  - для каждого запроса гененрируется уникальный идентификатор на стороне сервера с использованием GUID v7. Он будет использоваться далее как основной ижентификатор. Однако в самой базе должен использоваоться int идетификатор в качестве Primary Key, а GUID v7 должен быть просто одним из полей.
  - в блоб зранилище (seaweedfs) создается папка с именем, соответствующим GUID v7, и в эту папку сохраняются все изображения, связанные с этим запросом. Похожая вещь уже есть в ьекущей реализации на стороне ProcessingService. Нужно ее упростить немного, сделать короче путь. В остальном подход такой же - для каждого запроса есть уникальная папка для любых блобов, связанных с этим запросом. Имена блобов должны сгенерироваться автоматичски на основе GUID v7. Возможно стоит добавить какой-то префикс к именам, чтобы отличить эти блобы от других, которые могут быть в блоб хранилище, и чтобы было проще их идентифицировать и удалять при необходимости.
  - в базу данных сохраняется запись с данными запроса, включая описание, ссылку и информацию об изображениях (например, их имена или пути в блоб хранилище). Также сохраняется статус запроса, который изначально устанавливается в "ожидает обработки" или что-то подобное. Так же нужно сохранить как-то связь именно с теми изображенями, которые загрузил пользователь. Возможно в директории блоб-хранилища стоит завести поддиректорию именно для пользовательских изображений или каким-то образом помечать эти изображения, чтобы потом было проще их идентифицировать и удалять, если это будет необходимо (может быть с помощью префикса).
  - после сохранения данных в базу данных, API-сервис отправляет запрос Processing Servie для обработки. Эта часть уже реализована с помощью outbox pattern. Изучи ее и внесии изменения в дата-контракт.
  - пользователь получает ответ от API-сервиса с подтверждением того, что запрос был успешно получен и находится в очереди на обработку. В ответе также должен быть уникальный идентификатор запроса (GUID v7), который может быть использован для отслеживания статуса запроса и получения результатов анализа в будущем.
5. Обработка на стороне процессинг сервиса в целом остается прежней. Однако необходимо внести изменения в ту часть, что отвечает за взаимодействие с Claude API. На данный момент Claude API возвращает JSON-объект в ответе, который мы десириализуем. Это очень сложно. Такое решение тяжело поддерживать. Вместо этого мы должны получать от Claude лишь текст. Однако это должен быть форматированный ответ в markdown формате. Этот ответ должен быть структурирвоан строго по определенным правилам. Далее сам ответ нужно сохранить в базу и изменить статус запроса на "обработано". Важно, что мы не должны сохранять весь текст ответа в одном поле. Вместо этого сохранить его в соответствующей папке в блоб-хранилище, а в базе данных сохранять только ссылку на этот файл. Желательно использовать зарезирвированное имя для этого файла, например "ai-analysis-result.md". Для лучшей производительности можно дополнительно сгененрирваоть pdf-версию этого же ответа и сохранять ее в той же папке, что и markdown-файл.
