В рамках текущего проекта мы разрабатываем приложение, которое позволяет обычным пользователям анализировать объявления о продаже автомобилей и выявлять потенциальные риски, искать нестыковки в данных и предоставлять рекомендации по покупке. Более подробная информация изложена в директории `docs`.

На данный момент уже реализован прототип приложения, который выполняет проверку end-to-end. Однако, текущая реализация получилась запутанной и сложной для понимания. Поэтому мы планируем провести рефакторинг кода, чтобы улучшить его структуру и читаемость. А также улучшить user experience, чтобы сделать приложение более удобным и интуитивно понятным для пользователей.

Мы будем внедрять изменения с верху вниз, начиная с улучшения пользовательского интерфейса и затем переходя к оптимизации внутренней логики приложения. Это позволит нам обеспечить более плавный и приятный опыт для пользователей, а также повысить эффективность работы приложения в целом.

Изучи текущий код пользовательского интерфейса. На данный момент есть 2 страницы:
- главная страница, которая содержит кнопка входа на сайт с помощью Google аккаунта
- dashboard, который необходим для ввода информации и отображения результатов анализа объявлений о продаже автомобилей.

Необходимо избавиться от этого и перейте к единой странице, которая будет объединять функции обеих страниц. На этой странице будет реализована возможность входа через Google аккаунт. Эта часть функционала должна располагаться в центральной части страницы. Неавторизованный пользователь не должен иметь возможность анализировать объявления, а только видеть кнопку входа. После успешной авторизации пользователь будет перенаправлен на ту же страницу, но уже с доступом к функционалу анализа объявлений. Т.е. область с кнопкой входа будет заменена на более обширный интерфейс для ввода информации и отображения результатов анализа, а так же просмотра истории анализа пользователя.

Область для ввода данный должна состоять из нескольких компоенентов:
1. многострочный текстовый ввод для вставки текста объявления о продаже автомобиля. Он должен поддерживать базовое форматирвоание текста, такое как жирный шрифт, курсив и подчеркивание, чтобы пользователи могли выделять важные части объявления, списки и другие элементы форматирования для лучшей организации информации. Вероятнее всего подойдет какой-то markdpwn редактор, который поддерживает эти функции. Это позволит пользователям более эффективно структурировать информацию в объявлениях и выделять ключевые моменты для анализа.
2. кнопка для прикрпления изображений. Это необходимо, чтобы пользователь мог прикрепить фото автомобиля или какие-то скриншоты объявления, которые могут быть полезны для анализа. Прикрепленные изображения должны отображаться в виде миниатюр под текстовым вводом, и пользователи должны иметь возможность удалять их при необходимости, а так же просматривать их в полном размере при клике на миниатюру.
3. кнопка для прикрепления ссылки на сайт otomoto.pl. На данный момент мы поддерживаем только этот сайт, т.к. можем его распарсить и извлекать информацию для анализа. Пользователь может прикрепить только одну ссылку, и она должна быть валидной. Новая ссылка заменяет предыдущую, если она уже была прикреплена. Прикрепленная ссылка должна отображаться под текстовым вводом, и пользователи должны иметь возможность удалять ее при необходимости.

По факту же дата-контракт, который будет уходить с пользовательского интерфейса на бэкенд для анализа, будет включать в себя следующие поля:
- `description`: текс, который пользователь вставляет в многострочный текстовый ввод. Он должен сохранить форматирование оригинального текста в формате markdown, чтобы бэкенд мог правильно его интерпретировать и анализировать.
- `images`: коллекция изображений, которые пользователь прикрепляет. Максимальное количество изображений 5 (должно конфигурироваться через настройки сервиса). Максимальный размер изображения - 2560 кб (также должно конфигурироваться через настройки сервиса).
- `link`: текстовое поле со ссылкой. В целом это поле обезличино в датаконтракте. Но на бэкенде мы будем проверять, что ссылка действительно ведет на сайт otomoto.pl и извлекать из нее необходимую информацию для анализа. Если по каким-то причинам пришла ссылка на другой сайт, то мы будем тгнорировать ее и обрабатывать объявление только на основе текста и изображений, если они есть.

На этой же странице необходимо предусмотреть вывод истории анализа пользователя в виде списка с постраничным разбиением (не более 5 аналищов на страницу). При нажатии на элемент списка результат анализа будет подгружаться через отдельный запрос и отображаться в виде модального окна с затемненным фоном вокруг.

Итак, на текузем этапе необходимо внести изменения исключительно в пользовательский интерфейс. Логика работы приложения и взаимодействия с бэкендом будет уточчнена и реализована на следующем этапе. Важно, чтобы текущая реализация была чистой, понятной и легко поддерживаемой, чтобы в дальнейшем было проще добавлять новые функции и улучшать существующие.
