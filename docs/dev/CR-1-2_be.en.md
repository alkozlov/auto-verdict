Our next task is to update the API service and the database model.

We have already made changes to the user interface. Now we need to update the API service. Conceptually, we are not changing the behavior of the API service. The sequence should be as follows:
1. The user sends a POST request to the endpoint with the listing data (text, images, link).
2. The API service accepts the request.
3. Request validation:
  - the description must be provided — it is a required field
  - images are optional, but if present there must be no more than 5 and each image must not exceed 2560 KB
  - the link is optional, but if present it must be valid and point to otomoto.pl
4. If validation passes, the API service saves the user request data to the database.
  - a unique server-side identifier is generated for each request using GUID v7. It will be used going forward as the primary identifier. However, the database itself should use an `int` as the Primary Key, with GUID v7 being just one of the fields.
  - a folder named after the GUID v7 is created in blob storage (SeaweedFS), and all images associated with that request are saved into this folder. Something similar already exists in the current implementation on the ProcessingService side. It needs to be simplified a bit and the path shortened. Otherwise the approach is the same — each request has a unique folder for any blobs associated with it. Blob names should be generated automatically based on GUID v7. It may be worth adding some kind of prefix to the names to distinguish these blobs from others that may be in the blob storage and to make them easier to identify and delete when necessary.
  - a record is saved to the database with the request data, including the description, link, and information about the images (e.g. their names or paths in blob storage). A request status is also saved, initially set to "pending" or something similar. The association with the specific images uploaded by the user must also be stored. It may make sense to create a subdirectory within the blob storage directory specifically for user-uploaded images, or to mark these images in some way so that they are easier to identify and delete if necessary (perhaps with a prefix).
  - after saving data to the database, the API service sends a request to the Processing Service for processing. This part is already implemented using the outbox pattern. Review it and update the data contract.
  - the user receives a response from the API service confirming that the request was successfully received and is queued for processing. The response should also include the unique request identifier (GUID v7), which can be used to track the request status and retrieve analysis results in the future.
5. Processing on the Processing Service side remains the same in general. However, changes are needed in the part responsible for interacting with the Claude API. Currently, Claude API returns a JSON object in the response that we deserialize. This is overly complex and hard to maintain. Instead, we should receive only text from Claude — but it should be a formatted response in markdown format. This response must be structured according to strict rules. The response itself should then be saved to the database and the request status changed to "processed". Importantly, we must not save the entire response text in a single database field. Instead, save it to the corresponding folder in blob storage and store only a reference to that file in the database. It is preferable to use a reserved filename for this file, for example `ai-analysis-result.md`. For better performance, a PDF version of the same response can additionally be generated and saved in the same folder as the markdown file.
6. It also makes sense to clean up the database by removing tables that are no longer needed.
